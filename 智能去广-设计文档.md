# 智能去广功能设计文档

## 一、需求概述

在播放器中新增 **AI 辅助去广功能**,允许用户反馈"这个视频有广告",App 将视频 URL/片段信息提交给 AI 分析,AI 输出可能的广告规则(hosts/regex),用户可编辑、启用/禁用这些规则。

同时,为应对 **APP 默认规则可能误判** 的情况,在管理界面中**分区展示 AI 识别的规则和 APP 默认规则**,允许用户临时禁用某条默认规则。

## 二、核心设计

### 2.1 数据层

#### 2.1.1 用户自定义规则 (`UserAdRule`)

**已实现**,存储 AI 识别后用户编辑/添加的规则,持久化在 SharedPreferences (`user_ad_rules`)。

```java
class UserAdRule {
    String id;              // UUID,唯一标识
    String name;            // 规则名称(用户可编辑)
    List<String> hosts;     // 广告域名列表
    List<String> regex;     // URL 正则列表
    List<String> exclude;   // 排除正则列表
    boolean enabled;        // 是否启用
    String source;          // 来源标识:"ai" 或 "manual"
    long createdAt;         // 创建时间戳(ms)
    
    Rule toRule();          // 转为 Rule 对象(供 RuleConfig 合并)
}
```

**新增字段**(本次实现):
- `source`: 标识规则来源,`"ai"` 表示 AI 识别,`"manual"` 表示用户手动添加

#### 2.1.2 默认规则禁用列表 (`DisabledDefaultRuleStore`)

**新建**,持久化"被用户临时禁用的默认规则 ID"集合,存储在 SharedPreferences (`disabled_default_rule_ids`)。

```java
public class DisabledDefaultRuleStore {
    private static final String PREF_KEY = "disabled_default_rule_ids";
    
    // 加载禁用 ID 集合
    public static Set<String> load();
    
    // 保存禁用 ID 集合
    public static void save(Set<String> ids);
    
    // 添加禁用
    public static void disable(String ruleId);
    
    // 移除禁用
    public static void enable(String ruleId);
    
    // 判断某规则是否被禁用
    public static boolean isDisabled(String ruleId);
}
```

**Rule ID 计算规则**(用于唯一标识一条默认规则):
```java
String computeRuleId(Rule rule) {
    // 将 name + hosts + regex + exclude 序列化为稳定的 JSON,计算 MD5
    JsonObject obj = new JsonObject();
    obj.addProperty("name", rule.getName());
    obj.add("hosts", gson.toJsonTree(rule.getHosts()));
    obj.add("regex", gson.toJsonTree(rule.getRegex()));
    obj.add("exclude", gson.toJsonTree(rule.getExclude()));
    String json = gson.toJson(obj);
    return MD5(json);
}
```

#### 2.1.3 RuleConfig 合并逻辑修改

**现有**:
```java
private void merge() {
    List<Rule> rules = new ArrayList<>(VodConfig.get().getRules());
    rules.addAll(LiveConfig.get().getRules());
    rules.addAll(UserAdRuleStore.toRules());
    this.rules = rules;
    // ...
}
```

**修改后**:过滤掉被禁用的默认规则
```java
private void merge() {
    Set<String> disabledIds = DisabledDefaultRuleStore.load();
    
    // 添加默认规则时,跳过被禁用的
    List<Rule> defaultRules = new ArrayList<>(VodConfig.get().getRules());
    defaultRules.addAll(LiveConfig.get().getRules());
    List<Rule> enabledDefaults = defaultRules.stream()
        .filter(r -> !disabledIds.contains(computeRuleId(r)))
        .collect(Collectors.toList());
    
    List<Rule> rules = new ArrayList<>(enabledDefaults);
    rules.addAll(UserAdRuleStore.toRules());
    this.rules = rules;
    // ads 同理...
}
```

### 2.2 Service 层

#### 2.2.1 AI 去广检测服务 (`AiAdDetectionService`)

**已在前面任务中创建**,负责:
1. 收集视频播放上下文(URL、m3u8 内容、播放时长)
2. 构建 prompt 提交给 AI API
3. 解析 AI 返回的规则建议

**Prompt 设计要点**(参考 OmniBox 算法思路):
```
你是一个 HLS 视频广告检测专家。给定以下信息:
- 视频播放 URL
- m3u8 播放列表内容
- DISCONTINUITY 分块信息
- 各分块的分片数、总时长

请分析可能的广告片段,输出:
1. 广告域名列表(hosts)
2. 广告 URL 正则(regex)
3. 排除正则(exclude,避免误杀)

**关键判断依据**(参考 OmniBox 指纹算法):
- DISCONTINUITY 标记前后的分块,编码可能不同
- 短时长块(< 30秒)且跨域 → 高概率广告
- URL 中包含 ad/adv/preroll/vast 等关键词
- 域名与主视频域名不同
```

#### 2.2.2 规则 ID 计算工具

在 `RuleConfig` 或新建 `RuleIdUtil` 中实现:
```java
public class RuleIdUtil {
    public static String computeRuleId(Rule rule) {
        JsonObject obj = new JsonObject();
        obj.addProperty("name", rule.getName());
        obj.add("hosts", App.gson().toJsonTree(rule.getHosts()));
        obj.add("regex", App.gson().toJsonTree(rule.getRegex()));
        obj.add("exclude", App.gson().toJsonTree(rule.getExclude()));
        return MD5.encode(App.gson().toJson(obj));
    }
}
```

#### 2.3.2 去广规则管理界面 (`AdRuleManageDialog`)

**布局设计**(两个分区):

```
┌─────────────────────────────────┐
│  去广规则管理                     │
├─────────────────────────────────┤
│  [+ 新建规则]  [全部导出]  [导入] │
├─────────────────────────────────┤
│  ▼ AI 识别规则 (3)               │
│    ✓ 规则1: 某某广告域名          │
│    ✗ 规则2: 预加载广告            │
│    ✓ 规则3: ...                  │
├─────────────────────────────────┤
│  ▼ APP 默认规则 (12)             │
│    ✓ 默认1: 通用广告域名          │
│    ✗ 默认2: 某CDN(已禁用)        │  ← 可点击禁用/启用
│    ✓ 默认3: ...                  │
└─────────────────────────────────┘
```

**两个分区的差异**:
- **AI 识别规则**:来自 `UserAdRuleStore`,可编辑/删除/启用/禁用
- **APP 默认规则**:来自 `VodConfig/LiveConfig`,**只读**,但可"临时禁用"(实际是加入禁用列表)

**Adapter 实现**:
```java
class AdRuleAdapter {
    enum RuleType { AI_RULE, DEFAULT_RULE }
    
    static class RuleItem {
        RuleType type;
        String id;              // AI规则用 UserAdRule.id, 默认规则用 computeRuleId(rule)
        String name;
        boolean enabled;        // AI规则用 UserAdRule.enabled, 默认规则查 DisabledDefaultRuleStore
        boolean editable;       // AI规则=true, 默认规则=false
    }
    
    // 点击事件:
    // - AI规则 → 打开编辑 Dialog
    // - 默认规则 → 弹出确认"临时禁用此规则?"
}
```

#### 2.3.3 规则编辑 Dialog (`AdRuleEditDialog`)

**仅用于 AI 规则**,允许编辑 name/hosts/regex/exclude,保存到 `UserAdRuleStore`。

**字段**:
- 规则名称(单行文本)
- 广告域名(多行文本,每行一个)
- URL 正则(多行文本,每行一个)
- 排除正则(多行文本,可选)

## 三、交互流程

### 3.1 用户反馈"有广告"流程

```
[播放中] 用户点击"有广告"按钮
  ↓
[收集上下文] URL、m3u8、播放时长、DISCONTINUITY 块信息
  ↓
[调用 AI API] 提交 prompt,等待响应(显示 loading)
  ↓
[解析响应] AI 返回 { name, hosts[], regex[], exclude[] }
  ↓
[保存] 创建 UserAdRule,enabled=true, source="ai", 存入 UserAdRuleStore
  ↓
[刷新] RuleConfig.invalidate()
  ↓
[跳转] 打开 AdRuleManageDialog,高亮新增的规则
```

### 3.2 默认规则禁用/启用流程

```
[管理界面] 用户点击某条"APP 默认规则"
  ↓
[确认弹窗] "临时禁用此规则?可能导致部分广告无法过滤,可随时重新启用"
  ↓
[用户确认] 
  ↓
[禁用] DisabledDefaultRuleStore.disable(ruleId)
  ↓
[刷新] RuleConfig.invalidate() + Adapter.notifyItemChanged()
  ↓
[UI更新] 该规则显示为灰色/禁用图标
```

## 七、技术细节与注意事项

### 7.1 默认规则 ID 稳定性

**问题**:配置文件更新后,同一条规则的 JSON 顺序可能变化,导致 MD5 不同。

**解决**:序列化时对 hosts/regex/exclude 数组排序后再计算 MD5:
```java
obj.add("hosts", gson.toJsonTree(sorted(rule.getHosts())));
```

### 7.2 UI 性能

默认规则可能有几十上百条,Adapter 需优化:
- 使用 DiffUtil 计算增量更新
- 分区折叠/展开(初始只展开 AI 规则)

### 7.3 AI API 调用

- 需配置超时(30s)和重试(1次)
- 解析失败时友好提示,不崩溃
- API Key 从 Setting 读取,未配置时按钮不显示

### 7.4 参考 OmniBox 的启发

虽然本次不实现 ffprobe,但在 AI prompt 中融入 OmniBox 的思路:
- 强调 DISCONTINUITY 块的重要性
- 提示短块(<30s)且跨域的可疑性
- 避免仅靠 URL 关键词(可能误杀)
