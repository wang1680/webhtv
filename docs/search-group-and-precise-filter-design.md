# 搜索分组与精准过滤设计

> 状态：✅ 已实施并完成双端构建/测试验证
> 日期：2026-07-22
> 范围：手机版 `mobile` 与电视版 `leanback` 的影视搜索入口和搜索结果页
> 实施结果：TV 分组入口已改为确认键可见菜单；双端已接入结果分组筛选与精准标题过滤

## 1. 背景与现状判断

### 1.1 当前搜索链路

```text
手机版 SearchActivity / SearchFragment
电视版 SearchActivity
        │
        ├─ 选择搜索范围：全部 / 当前站 / 指定分组
        ▼
手机版 CollectFragment
电视版 CollectActivity
        │
        ▼
SiteViewModel.searchContent(List<Site>, keyword, quick=false)
        │
        ▼
SearchTask -> SiteApi.searchContent()
        │
        ├─ 多站并发结果：getSearch()，按站点逐批返回
        └─ 单站翻页结果：getResult()
```

搜索结果页左侧/顶部的 `Collect` 列表表示“全部 + 各站点”，右侧为当前选中范围内的 `Vod` 列表。

### 1.2 手机版已有能力

手机版当前已经具备两类分组能力：

1. **搜索前限定范围**：`SearchFragment` 的范围菜单支持“全部、当前站、指定分组”，并将 `siteKey/group` 传给 `CollectFragment`，从请求源头减少参与搜索的站点。
2. **搜索后分组筛选**：全局搜索进入 `CollectFragment` 后，可通过 `action_group` 在已返回结果中切换站点分组，不重新发起网络搜索。

`CollectFragment` 使用 `mAllCollectItems` 保存未分组过滤的站点结果，然后通过 `mFilterGroup` 派生当前展示列表。这部分是电视版结果页最适合参考的实现。

### 1.3 电视版实际现状

当前代码不是完全没有分组能力，而是存在两个问题：

1. `SearchActivity` 已经支持分组搜索范围，但入口隐藏在 `searchScope` 的**长按**操作中；普通确认键只在“全部/当前站”之间切换。对遥控器用户来说几乎不可发现，因此产品体验上等同于“没有分组搜索”。
2. `CollectActivity` 只支持搜索前传入的 `group`，没有手机版结果页已有的“搜索后按站点分组即时筛选”。

因此本设计把“电视版分组搜索”拆成两个补齐项：

- 搜索页：让分组范围选择可见、可发现。
- 结果页：增加无需重搜的分组二次筛选。

## 2. 目标

1. 电视版使用遥控器确认键即可选择“全部、当前站、指定分组”，不依赖长按隐藏手势。
2. 电视版全局搜索结果页具备与手机版一致的站点分组二次筛选。
3. 手机版和电视版搜索结果页都增加“精准过滤”开关，仅展示标题与搜索词高度相关的结果。
4. 分组过滤与精准过滤可以同时启用，并按 AND 关系组合。
5. 所有原始搜索结果必须保留在内存中，关闭过滤后立即恢复，不重新请求网络。
6. 不改变搜索接口、不影响站点健康统计、不引入第三方相似度依赖。

## 3. 非目标

本期不处理以下内容：

- 不做语义别名匹配，例如“哈利波特”和“Harry Potter”。
- 不做拼音、谐音、AI 标题理解。
- 不使用演员、导演、简介、类型或备注参与精准匹配，只匹配 `Vod.getName()`。
- 不对搜索结果去重。
- 不改变站点原始结果排序，也不按相似度重新排序。
- 不增加可配置阈值；第一版采用内置保守规则。
- 不把过滤下沉到爬虫、`SiteApi` 或服务端。

## 4. 核心设计原则

### 4.1 原始数据与展示数据分离

精准过滤是可逆的界面过滤，不能直接删除 `Result.getList()` 或 `Collect.getList()` 中的原始数据。

每个结果页维护两层状态：

```text
原始层 Raw
  各站点已经返回的全部 Vod
  各站点翻页页码
  站点原始顺序

展示层 Projection
  站点分组过滤
  AND 精准标题过滤
  -> 全部聚合项
  -> 可见站点项
  -> 当前右侧结果
```

所有网络结果先写入原始层，再根据当前过滤状态生成或增量更新展示层。切换过滤只重新投影，不重新搜索。

### 4.2 精准过滤放在结果聚合层

精准过滤应在 `CollectFragment/CollectActivity` 接收到结果后执行，而不是放在 `SearchTask`、`SiteApi` 或 `SiteViewModel` 中，原因如下：

- 用户可以随时关闭过滤并恢复已经加载的数据。
- `SiteHealthStore.recordSearch()` 继续记录站点真实返回数量，不会因为本地相关性规则而错误降低站点健康度。
- 搜索进度仍以完成站点数计算，与过滤结果多少无关。
- 单站翻页和多站并发结果可以使用同一过滤逻辑。

### 4.3 过滤不改变排序

通过精准规则的条目保持站点返回顺序；“全部”聚合保持当前站点顺序：

- 动态模式：按站点响应顺序。
- 固定模式：按配置站点顺序。

## 5. 电视版分组搜索设计

### 5.1 搜索页范围选择

#### 当前问题

`searchScope` 单击仅切换“全部/当前站”，长按才显示完整分组菜单，缺乏可发现性；当前弹窗宽度固定为 `136dp`，长分组名还可能截断。

#### 调整方案

- `searchScope` 的普通确认键直接打开完整范围菜单。
- 菜单选项顺序：
  1. 全部
  2. 当前站
  3. 配置中解析出的所有站点分组
- 长按可以保留为打开同一菜单的兼容入口，但不再承载独占功能。
- 按钮文本始终显示当前范围：`全部 / 当前站 / 分组名`。
- 建议增加下拉箭头或选中态，明确它是可展开选择器。
- 弹窗宽度根据最长文本动态计算；高度最多显示 8 项，超过后滚动。
- 弹窗打开后默认聚焦当前选中项。
- “当前站”不可搜索时沿用现有提示，并保持原选择不变。

#### 请求行为

- 选择“全部”：`siteKey=""`、`group=""`。
- 选择“当前站”：传当前 `siteKey`、`group=""`。
- 选择分组：`siteKey=""`、传对应 `group`。

这部分是搜索前限定范围，会真实减少 `mSites` 和并发请求数量。

### 5.2 电视版结果页分组二次筛选

在 `activity_collect.xml` 顶部操作区增加可聚焦的分组按钮，位置建议为：

```text
搜索结果标题                    [分组：全部] [精准] [布局]
```

行为：

- 仅当本次是全局搜索，且存在至少一个分组时显示。
- 当前站搜索或搜索前已经限定分组时隐藏，避免产生含义冲突。
- 确认键打开“全部 + 分组列表”。
- 切换分组只过滤已经加载的站点和结果，不发起新请求。
- 后续异步返回的站点结果仍写入原始层；只有属于当前分组的结果进入展示层。
- 当前选中站点在新分组中仍存在时保留选择，否则回退到“全部”。
- 切换后右侧结果滚动到顶部。

电视端实现可参考手机版 `CollectFragment` 的 `mAllCollectItems`、`mFilterGroup`、`getFilteredCollectItems()` 流程，但不能继续把 Adapter 数据当作唯一真源。

## 6. 双端精准搜索过滤设计

### 6.1 交互形态

精准过滤采用一个结果页开关，默认关闭，并记住用户上次选择：

- 手机版：在 `menu_collect.xml` 增加 `action_precise`。建议使用图标或短文本“精准”，避免与分组文本、布局按钮共同挤压标题。
- 电视版：顶部增加可聚焦的 `preciseFilter` 按钮，选中态表示开启。
- 状态通过 `Setting.isSearchPrecise()` / `Setting.putSearchPrecise(boolean)` 保存。
- 搜索词归一化后长度小于 2 时不允许开启，并提示“关键词过短，无法精准过滤”。
- 开启/关闭后立即对已加载结果重新投影，不重新请求。
- 开启后无结果时显示明确空状态：`暂无高匹配结果，可关闭精准过滤查看全部`。

默认关闭的原因：任何启发式规则都可能产生漏召回；精准过滤适合作为用户主动收窄结果的工具，而不是静默改变既有搜索行为。

### 6.2 匹配字段

第一版只使用：

```java
Vod.getName()
```

不使用 `typeName`、`remarks`、`actor`、`director`、`content`，否则“标题不相关、详情文本碰巧包含关键词”的结果会重新进入列表，失去精准过滤意义。

### 6.3 文本归一化

新增共享纯逻辑类，建议：

```text
app/src/main/java/com/fongmi/android/tv/utils/SearchResultFilter.java
```

归一化步骤：

1. 空值转空字符串。
2. `Trans.t2s()` 将繁体转为简体，与现有搜索任务保持一致。
3. Unicode NFKC 归一化，统一全角/半角字符。
4. 使用 `Locale.ROOT` 转小写。
5. 将明确的季/部表达标准化为数字，例如：
   - `第二季`、`第2季`、`Season 2`、`S02` -> `2`
   - 只处理带“季/部/Season/S”语义的数字，不删除普通片名数字。
6. 标点和空白统一处理：
   - 保留一份 token 文本，供短英文词整词判断。
   - 生成一份仅含 Unicode 字母和数字的 compact 文本，供精确、包含和编辑距离比较。
7. 不删除普通年份或片名尾部数字，避免把《无间道2》和《无间道3》视为同一标题。

不直接复用 `MediaTitleParser.cleanTitle()`：该类为了 TMDB/弹幕检索会主动移除年份、季号和部分尾部数字，适合生成搜索候选词，但对“精准结果过滤”过于激进，容易合并不同续集。

### 6.4 判定顺序

按以下顺序判定，任一命中即保留：

#### A. 完全匹配

```text
normalizedTitle == normalizedKeyword
```

优先级最高。

#### B. 标题包含搜索词

```text
normalizedTitle contains normalizedKeyword
```

限制：

- 中文或包含中文的关键词：归一化长度至少 2。
- 纯英文/数字关键词长度至少 3 时允许 compact 包含。
- `IT`、`UP`、`US` 等短英文标题使用 token 整词匹配，避免 `IT` 误命中 `Little Women`。
- 不默认采用“搜索词包含结果标题”的反向包含，因为用户输入了更具体的续集、年份或版本时，反向包含容易把基础标题错误保留。

#### C. 高相似度模糊匹配

仅在未命中 A/B 时执行标准库实现的 Levenshtein 编辑距离：

```text
similarity = 1 - editDistance(title, keyword) / max(titleLength, keywordLength)
```

建议阈值：

- 含中文：`similarity >= 0.75`
- 纯拉丁字母/数字：`similarity >= 0.82`
- 两侧最大长度必须至少为 4。
- 长度差不得超过 `max(2, floor(maxLength * 0.2))`。
- 如果两边都含数字且数字序列不同，禁止模糊命中。
- 如果搜索词含数字、结果标题不含对应数字，也禁止模糊命中。

数字保护可避免：

- `无间道2` 误命中 `无间道3`
- `三体2023` 误命中 `三体2024`

### 6.5 示例

| 搜索词 | 结果标题 | 结论 | 原因 |
|---|---|---:|---|
| `庆余年` | `庆余年 第二季` | 保留 | 标题包含搜索词 |
| `庆余年2` | `庆余年 第二季` | 保留 | 季号标准化后完全匹配 |
| `三体` | `[腾讯] 三体 (2023) 4K` | 保留 | 归一化后包含 |
| `The Last of Us` | `The.Last.of.Us.S01` | 保留 | 标题包含搜索词 |
| `流浪地球2` | `流浪地求2` | 保留 | 数字一致且中文相似度达到阈值 |
| `长相思` | `长相守` | 过滤 | 三字标题不启用模糊匹配 |
| `无间道2` | `无间道3` | 过滤 | 数字序列冲突 |
| `IT` | `IT Chapter Two` | 保留 | 短英文整词匹配 |
| `IT` | `Little Women` | 过滤 | 不是整词匹配 |
| `沙丘2` | `沙丘` | 过滤 | 搜索词的数字信息缺失 |

### 6.6 结果投影规则

最终展示条件：

```text
siteMatchesSelectedGroup
AND
(!preciseEnabled OR SearchResultFilter.matches(keyword, vod.name))
```

“全部”项必须由过滤后的各站点列表重新聚合，不能继续引用原始总列表。

#### 固定站点顺序模式

- 保留配置中的站点行，即使当前精准过滤后该站点暂时为 0 条，以保持“固定模式”的既有语义和进度感知。
- 右侧列表为空时展示空状态。

#### 动态站点顺序模式

- 未开启精准过滤时，只有至少一条可见结果的站点才进入站点列表。
- 开启精准过滤后，已有原始结果但当前尚无高匹配条目的站点仍保留入口，以便继续翻页并展示加载或空状态。
- 关闭精准过滤后，之前只有低相关结果的站点可以重新出现。

## 7. 异步、翻页与状态一致性

### 7.1 多站并发结果

每个 `getSearch()` 回调按以下顺序处理：

1. 将完整 `Result.getList()` 写入对应站点的原始列表。
2. 根据当前分组判断站点是否可见。
3. 根据当前精准状态过滤本批条目。
4. 增量更新当前展示，或在过滤状态刚变化时执行完整投影。
5. 搜索进度无论本批是否有可见条目都正常递增。

### 7.2 单站翻页

翻页结果必须先追加到原始站点列表，再追加当前可见子集：

- 精准开启时，当前页中不匹配的条目只保存在原始层。
- 关闭精准过滤后，所有已加载页中的隐藏条目立即恢复。
- 页码保存到原始站点状态，不能只写到临时展示副本。
- 翻页请求发出时只记录“站点 + 待请求页”，收到该站点的非空结果后才提交页码；空结果、错误结果和错站结果均不推进页码。
- 同一结果页同一时间只允许一个单站翻页请求。切换分组、精准状态或选中站点不会清除该请求，也不会发起会取消它的新请求。
- 即使当前分页整批都被精准过滤隐藏，展示副本已有站点项时也要同步页码；切换站点恢复分页时始终优先读取原始层页码。
- 当前选中站点的某一页有原始数据但整页被精准过滤隐藏时，自动继续请求下一页，直到出现可见条目、到达明确总页数或服务端返回空页。
- 每个站点记录已接收页面的稳定条目签名；若未知总页数的站源重复返回相同非空页，则不追加重复数据、不提交页码，并停止该站点继续翻页，避免自动请求循环。
- 若过滤切换使动态站点项暂时消失，进行中的分页随后产生首批可见结果时，应重新创建该站点的展示投影，并在“全部”处于激活状态时同步追加右侧列表。

### 7.3 过滤切换时的选中项

- 优先按 `siteKey` 恢复原选中站点，不按旧 position 恢复。
- 原站点被分组隐藏或在动态模式下无可见结果时，回退到“全部”。
- 右侧列表滚动到顶部。
- 电视版如果当前焦点位于即将消失的条目，应把焦点移到“精准”按钮或首个可见结果，避免焦点丢失。

## 8. 建议代码改动范围

### 8.1 共享代码

```text
app/src/main/java/com/fongmi/android/tv/utils/SearchResultFilter.java
app/src/main/java/com/fongmi/android/tv/setting/Setting.java
app/src/main/res/values/strings.xml
app/src/main/res/values-zh-rCN/strings.xml
```

`SearchResultFilter` 保持纯函数、无 Android View 依赖，便于 JVM 单元测试。

### 8.2 手机版

```text
app/src/mobile/java/com/fongmi/android/tv/ui/fragment/CollectFragment.java
app/src/mobile/res/menu/menu_collect.xml
app/src/mobile/res/layout/fragment_collect.xml（空状态，如采用）
```

重点：

- `mAllCollectItems` 是包含完整结果、站点顺序和分页页码的唯一原始层，Adapter 中的 `Collect` 仅是展示副本。
- 使用按 `siteKey` 保存的 `mVisibleItems` 缓存当前精准状态下的可见条目；多站回调和分页回调只过滤本批新增结果并追加缓存，避免每次结果到达都重新处理全部历史标题。
- 仅在精准开关变化时通过 `rebuildVisibleItems()` 全量重建缓存；分组切换只重新聚合已有缓存，不重复执行繁简转换、正则和编辑距离。
- `getFilteredCollectItems()` 从可见缓存复制列表并生成 Adapter 投影，分页和选中恢复始终按 `siteKey` 回到原始对象读取状态。

### 8.3 电视版

```text
app/src/leanback/java/com/fongmi/android/tv/ui/activity/SearchActivity.java
app/src/leanback/java/com/fongmi/android/tv/ui/activity/CollectActivity.java
app/src/leanback/java/com/fongmi/android/tv/ui/adapter/CollectAdapter.java
app/src/leanback/res/layout/activity_search.xml
app/src/leanback/res/layout/activity_collect.xml
```

重点：

- 搜索页普通确认键打开完整范围菜单。
- 结果页增加分组和精准按钮。
- 增加原始结果容器，不能继续只依赖 `CollectAdapter` 保存数据。
- 原始层保存权威页码；`scheduleCollect()` 切站恢复时优先读取原始对象，展示副本即使本页无可见条目也同步页码。
- 多站与分页回调共享站点投影追加逻辑：动态模式下，进行中的分页若在过滤切换后首次产生可见结果，可重新创建站点项，避免只更新“全部”而遗漏站点入口。
- 保留现有 TV 结果懒加载、滚动期间批量追加和图片加载优化；普通结果到达时优先增量更新，只有用户切换过滤时执行完整重建。

## 11. 风险与控制

### 11.1 误过滤有效结果

风险：别名、译名、缩写或短标题无法通过字面规则。

控制：

- 默认关闭并允许即时恢复。
- 第一版只采用保守阈值。
- 不把过滤结果写回缓存或数据库。
- 不影响原始网络数据和站点健康评分。

### 11.2 原始列表和展示列表混用

风险：切换过滤后数据丢失、翻页追加到临时副本、关闭过滤无法恢复。

控制：

- 原始层是唯一真源。
- Adapter 只保存展示投影。
- 所有网络回调先写原始层。
- 使用 `siteKey` 定位原始站点，不依赖当前 Adapter position。

### 11.3 电视版焦点回归

风险：顶部新增按钮、弹窗和动态隐藏站点可能导致焦点丢失或方向键死区。

控制：

- 明确顶部按钮顺序和 `nextFocus*`。
- 弹窗打开后聚焦当前项。
- 过滤后校验焦点 View 是否仍 attached/visible。
- 在电视真机或模拟器覆盖横向、纵向结果布局。

### 11.4 过滤性能

标题通常较短，Levenshtein 仅在精确和包含都失败后执行；对几百到一两千条结果成本可控。实现时应复用一次归一化后的 keyword，避免每个条目重复处理搜索词。第一版不需要后台线程或缓存体系，若真机测量出现明显卡顿再优化。

## 12. 实施顺序

1. 新增 `SearchResultFilter` 与单元测试，固定匹配规则。
2. 重整手机版结果页的原始层/展示层边界，并接入精准开关。
3. 参照手机版给电视版结果页增加原始结果容器和分组过滤。
4. 给电视版接入精准过滤，保留现有懒加载和滚动优化。
5. 将电视版搜索页范围按钮改为单击完整菜单。
6. 补齐双端空状态、字符串、无障碍描述和 TV 焦点规则。
7. 运行双 flavor 单测、构建和手工验证矩阵。

## 13. 验收标准

- TV 搜索页不使用长按也能选择任意站点分组。
- TV 全局搜索结果页可以在“全部/分组”之间切换，且不重新发起搜索。
- 手机版原有分组过滤行为不回归。
- 双端都能开启/关闭精准过滤，并恢复全部已加载结果。
- 精准过滤至少正确处理完全匹配、标题包含、季号等价、高相似度和数字冲突。
- 分组与精准同时开启时结果正确，后续异步结果和翻页结果遵循当前状态。
- 过滤不改变站点请求数量、搜索进度和站点健康统计。
- TV 遥控器焦点无丢失、无不可达控件、无方向键死区。
- 双端单元测试和 Debug 构建通过。

## 14. 建议结论

建议采用“**搜索前范围限制 + 搜索后可逆过滤**”的双层方案：

- 搜索前分组用于减少请求，解决效率问题。
- 搜索后分组用于快速查看已返回数据，解决浏览问题。
- 精准过滤作为默认关闭、可持久记忆的结果页开关，使用保守的“完全匹配 -> 标题包含 -> 高相似度”规则。
- 原始结果始终保留，所有过滤都只作用于展示投影。

该方案复用手机版已有分组思路，同时补齐电视端的可发现性、结果页一致性和可逆精准过滤，风险主要集中在结果状态分层与 TV 焦点，均可通过上述数据边界和测试矩阵控制。
